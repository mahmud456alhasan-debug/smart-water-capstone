\begin{abstract}
This report documents Week~4 Session~B on \textbf{integration and flow}: wiring a rainfall monitoring pipeline (weather client, SCS-CN processor, alerts, CSV storage) with OpenCode Desktop, then hardening errors and building a Streamlit dashboard. Work was done in \texttt{week4\_session\_b/} using Open-Meteo by default (no API key). Exercise~1 integrated modules with logging, retries, and boundary exceptions; Exercise~2 added \texttt{ConfigError}, validated configuration, and tested CONFIG ERROR / DEGRADED paths; Exercise~3 delivered a dashboard reading \texttt{data/monitoring\_log.csv}. Evidence: seven screenshots (OpenCode, terminal, Streamlit) and source in Appendix~\ref{app:main}--\ref{app:promptlog}.
\end{abstract}

\section{Introduction}
Integrated systems connect independent components through clear interfaces and error boundaries. This session practiced \textbf{staying in flow with AI}: one feature thread at a time (integrate, then errors, then UI), paste interfaces first, and verify with small terminal loops after each step. Week~4 Session~A refactored a single legacy script; Session~B builds a multi-module monitor aligned with earlier SCS-CN work (Week~3) but focused on orchestration and resilience.

\section{Objectives}
\begin{itemize}[noitemsep]
  \item Integrate weather fetch, rainfall processing, alerts, and CSV storage.
  \item Implement robust error handling (config, API, storage).
  \item Build a Streamlit monitoring dashboard.
  \item Document data flow and AI-assisted workflow in \texttt{prompt\_log.md}.
\end{itemize}

\section{Methods}

\subsection{Architecture}
Four components plus configuration:
\begin{enumerate}[noitemsep]
  \item \texttt{weather\_client.py} --- precipitation via Open-Meteo or OpenWeather (\texttt{OPENWEATHER\_API\_KEY})
  \item \texttt{rainfall\_processor.py} --- SCS-CN runoff (\texttt{process\_reading})
  \item \texttt{alert\_system.py} --- OK / WATCH / ALERT thresholds
  \item \texttt{storage.py} --- append rows to \texttt{data/monitoring\_log.csv}
  \item \texttt{config.py} --- \texttt{load\_settings()} from environment
\end{enumerate}
Orchestration: \texttt{main.py} \texttt{run\_once()}. Visualization: \texttt{dashboard.py} via \texttt{python3 -m streamlit run}.

\subsection{Exercise 1: Integration}
Prompt \texttt{prompt\_integration.txt} enhanced \texttt{main.py} and \texttt{weather\_client.py}: pipeline logging, \texttt{WeatherAPIError} $\rightarrow$ DEGRADED (exit~1), \texttt{StorageError} $\rightarrow$ WARNING (non-fatal), exponential backoff retries (\texttt{api\_max\_retries}).

\subsection{Exercise 2: Error handling}
Prompt \texttt{prompt\_errors.txt} added \texttt{ConfigError}, moved env parsing into \texttt{load\_settings()}, JSON parse guards, and \texttt{OSError} handling in storage. Tested \texttt{MONITOR\_LAT=abc}, \texttt{MONITOR\_LAT=999}, and \texttt{API\_TIMEOUT\_SEC=0.001}.

\subsection{Exercise 3: Dashboard}
Prompt \texttt{prompt\_dashboard.txt} produced a wide-layout Streamlit UI: metrics, color-coded alerts, line chart, recent table, empty-CSV handling.

\subsection{Exercise 4: Verification}
Checklist: cold-start \texttt{main.py}, failure paths, CSV growth, dashboard reflects latest row.

\section{Analysis and validation}

\textbf{Data flow:} A single \texttt{run\_once()} call exercises all components; CSV is the handoff to the dashboard---no duplicate API calls in the UI.

\textbf{Open-Meteo vs.\ lab template:} The handout mentions OpenWeather; implementation defaults to Open-Meteo (keyless prototyping) with optional OpenWeather via environment---consistent with Week~2 project choices.

\textbf{Resilience:} Failed API runs do not append corrupt rows when the pipeline exits before \texttt{append\_reading}. Config errors fail before network I/O.

\textbf{Flow discipline:} Separate prompts (\texttt{integration}, \texttt{errors}, \texttt{dashboard}) matched the lab's one-thread-per-feature guidance and reduced context loss.

\section{Discussion}
\begin{enumerate}[noitemsep]
  \item Exercise~1 already included retries; Exercise~2 added config validation that fixed import-time \texttt{ValueError} on bad \texttt{MONITOR\_LAT}.
  \item Multiple OpenCode screenshots for Exercise~2 reflect scroll length, not separate prompts.
  \item \texttt{python3 -m streamlit} is required when the Streamlit CLI is not on PATH.
  \item Dashboard reads CSV only---appropriate separation for a monitoring view.
\end{enumerate}

\section{Problems encountered}
\texttt{streamlit: command not found} resolved with \texttt{python3 -m streamlit run}. First-time Streamlit email prompt required pressing Enter on a blank line. Pip reported a jupyterlab version conflict unrelated to this lab.

\section{Lessons learned}
Integrate with explicit boundary messages (\texttt{CONFIG ERROR} vs.\ \texttt{DEGRADED}). Test three failure modes after AI edits. Use module-form CLI invocations on Linux lab machines. Screenshot terminal and UI as evidence alongside \texttt{prompt\_log.md}.

\section{Conclusion}
Week~4 Session~B objectives were met: integrated rainfall monitor, hardened error paths, Streamlit dashboard, and documented AI workflow. Submitted: \texttt{week4\_session\_b/} sources, \texttt{prompt\_log.md}, and seven screenshots in \texttt{Lab-reports/}.
